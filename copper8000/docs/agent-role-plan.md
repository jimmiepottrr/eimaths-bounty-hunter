# Agent Role — Rollout Plan (Copper 8000)

แผนการเพิ่ม **role ใหม่ `agent`** (พนักงานขาย / นายหน้า) เข้าระบบจองราคาโลหะ Copper 8000
โดยแบ่งเป็นเฟส เพื่อให้แต่ละเฟสเล็ก รีวิวง่าย และไม่ทิ้งโค้ดที่ยังไม่ได้ใช้

## ทำไมต้องมี role `agent`

วันนี้ระบบมี 2 role — `user` (ลูกค้าที่จองราคา) กับ `admin` (ผู้ดูแล). ธุรกิจรับซื้อโลหะจริงมี
"พนักงาน/นายหน้า" ที่คอยช่วยลูกค้าจอง ดูรายการ และประสานงาน แต่ไม่ควรมีสิทธิ์ระดับแอดมิน
(แก้ราคา, อนุมัติสมาชิก, เปลี่ยนธีม/ภาษา). `agent` คือ role กลางระหว่าง `user` กับ `admin`.

## สรุป role หลังแต่ละเฟส

| Role    | สมัครเอง | ต้องรออนุมัติ | จองราคาได้ | จองแทนลูกค้า | คอนโซลแอดมิน |
|---------|----------|---------------|------------|--------------|--------------|
| `user`  | ได้       | ใช่ (จองไม่ได้จนกว่าจะอนุมัติ) | ได้ | — | ไม่ได้ |
| `agent` | **ไม่ได้ (แอดมินสร้างเท่านั้น)** | ไม่ต้อง (staff auto-approved) | **ไม่ได้ (ดูราคาอย่างเดียว)** | **ไม่มีความสามารถนี้** | ไม่ได้ |
| `admin` | ไม่ได้    | —             | ได้        | —             | ได้           |

## เฟส A — Foundation (เฟสนี้ ✅)

ทำให้ `agent` เป็น role ระดับเฟิร์สคลาสในทุกจุดที่ระบบอ้างถึง role อยู่แล้ว **โดยยังไม่ให้สิทธิ์ใหม่**
เป้าหมาย: agent ล็อกอินได้ และแอปแสดงตัวตนว่าเป็น "พนักงาน" อย่างถูกต้อง (แยกจากแอดมินและลูกค้า)

ขอบเขต:
- **Types** — ขยาย `User.role` เป็น `'user' | 'agent' | 'admin'` (`src/data/types.ts`)
- **DB** — เพิ่ม `'agent'` เข้า `role ENUM` ใน `server/schema.sql` + ไฟล์ย้าย
  `server/migrate_agent_role.sql` (`ALTER TABLE`) ให้ DB บน VPS อัปเกรดได้โดยไม่เสียข้อมูล
- **Seed** — เพิ่มบัญชี agent สาธิตใน `src/data/seed.ts` (`agent@copper8000.co.th`, approved)
- **Display** — เพิ่มคีย์ i18n `auth.agentRole` (th/en/zh) + ป้าย role ของ agent ใน
  `Layout.tsx` (แถบหัว) และ `ProfilePage.tsx` (หน้าข้อมูลผู้ใช้)
- **Test** — smoke: agent ล็อกอินแล้วแถบหัวขึ้นป้าย "พนักงาน"

หมายเหตุการออกแบบ (เฟส A):
- agent ถูกสร้างโดยแอดมิน/seed เท่านั้น — **ไม่มี** self-signup (คงเส้นทาง signup เป็น `user` เหมือนเดิม)
- agent เป็น staff จึง auto-approved — ไม่ต้องผ่านคิวอนุมัติ (คิวอนุมัติกรอง `role='user'` อยู่แล้ว
  agent จึงไม่หลุดเข้าคิว โดยไม่ต้องแก้ query)
- guard เดิม `role !== 'admin'` กัน agent ออกจากคอนโซลแอดมินอยู่แล้ว — เฟส A ไม่แตะ guard
- จนกว่า UI สร้าง agent จะมาในเฟส C แอดมินสร้าง agent บน backend จริงผ่าน SQL/`migrate` ไปก่อน

## เฟส B — แอดมินสร้าง agent + หน้า login พนักงานแยก (เฟสนี้ ✅)

ตามที่ Jim กำหนด: **agent สมัครเองไม่ได้ ต้องให้แอดมินสร้างเท่านั้น** และ **agent ไม่มีความสามารถจองแทนลูกค้า**

ขอบเขต:
- **แอดมินสร้าง/จัดการ agent** — แท็บ "พนักงาน" ในคอนโซลแอดมิน: สร้าง (email/ชื่อ/เบอร์/รหัสผ่าน),
  ดูรายการ, ลบ. agent ที่สร้างเป็น `role='agent'`, `approved=1` อัตโนมัติ
  - backend: `admin.php` เพิ่ม `?view=agents`, `action=create_agent`, `action=delete_agent`
  - service: `listAgents` / `createAgent` / `deleteAgent` (ทั้ง http + mock adapter)
- **หน้า login พนักงานแยกต่างหาก** — route `/agent-login` (`AgentLoginPage`) แยกจากหน้า login ผู้ใช้
  - ถ้าคนที่ล็อกอินไม่ใช่ agent/admin → เคลียร์เซสชันแล้วแจ้งเตือน (เพิ่ม `clearSession()` ใน store)
  - มีลิงก์ไป-กลับระหว่างหน้า login ผู้ใช้กับหน้า login พนักงาน
- **agent จองไม่ได้** — หน้า `สินค้า` แสดง "มุมมองพนักงาน — ดูราคาอย่างเดียว" และแตะจองไม่ได้
  (ครอบคลุมทั้งจองให้ตัวเองและจองแทนลูกค้า — ตัดความสามารถจองแทนลูกค้าออกตามที่ Jim สั่ง)

## เฟส C — จัดการ agent เพิ่มเติม + แดชบอร์ด (ยังไม่ทำ)

- **หน้าแอดมิน**: ผูกลูกค้าให้ agent ดูแล / แก้ไขข้อมูล agent
- **แดชบอร์ด agent**: สรุปลูกค้าในความดูแล
- (ตัวเลือก) เชื่อม CRM / LINE OA สำหรับติดตามลูกค้า

## การทดสอบ (เฟส A)

1. `cd copper8000 && npm ci` (ถ้ายังไม่ได้ลง deps)
2. `npm run build` — `tsc --noEmit && vite build` ต้องผ่าน (การขยาย union `role` ต้องคอมไพล์ครบทุกจุด)
3. `npm test` — Playwright smoke เดิมเขียว + เคส agent ใหม่ผ่าน
4. โหมดสาธิต: `npm run dev` → ล็อกอิน `agent@copper8000.co.th` / `agent1234` → แถบหัวและ
   `/#/profile` ขึ้น "พนักงาน"; ยืนยันว่า `/#/admin` ยัง redirect agent ออก และ agent ไม่โผล่ในคิวอนุมัติ
